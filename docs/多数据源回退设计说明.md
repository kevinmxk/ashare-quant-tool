# 多数据源回退设计说明

## 当前回退顺序

当 `ASHARE_QUANT_PROVIDER=auto` 时，系统按以下顺序尝试：

1. `AKShare`
2. `Tushare`
3. `BaoStock`
4. `Mock`

其中前三个属于真实数据源，最后一个用于开发兜底。

## 为什么这样排序

### AKShare

- 更适合个人场景快速取数
- 对行情快照和分钟级延展能力更友好

### Tushare

- 更适合日频研究和结构化日度指标
- 但需要 `token`，且部分接口存在积分门槛

### BaoStock

- 更适合历史研究和补数
- 作为最后一层真实数据备份更稳妥

## 当前实现方式

系统增加了 `CompositeMarketDataProvider`：

- `list_universe` 失败时切换下一个源
- `get_quote` 失败时切换下一个源
- `get_daily_bars` 失败时切换下一个源

这意味着同一台机器上即使某个真实源暂时不可用，系统也不会直接报废。

## 迁移到新电脑时的建议

1. 默认先保留 `ASHARE_QUANT_PROVIDER=auto`
2. 如果你已经准备好 `Tushare token`，就写入 `.env`
3. 如果某个源单独调试，才显式指定：

- `ASHARE_QUANT_PROVIDER=akshare`
- `ASHARE_QUANT_PROVIDER=tushare`
- `ASHARE_QUANT_PROVIDER=baostock`

4. 如果只是先验证本地骨架，可以临时使用：

- `ASHARE_QUANT_PROVIDER=mock`

## 已知边界

- `Tushare` 的 `pro_bar`、`daily_basic` 可用性与账号权限有关
- `BaoStock` 更偏历史研究，`volume_ratio` 这类字段不完整
- 三个真实源之间字段口径不完全一致，因此评分层目前采用了“尽量统一、缺失容忍”的策略
